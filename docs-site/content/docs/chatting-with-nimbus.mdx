Chat is where you drive Nimbus. You ask in plain language; Nimbus decides which cloud tools to use, shows its reasoning and each tool call live, and returns an answer, often having read real inventory, telemetry, logs, or cost along the way.

## The diagnose workflow

When you ask why something is broken or slow, Nimbus follows a consistent path:

<Steps>
  <Step title="Read telemetry">
    Pulls live metrics (CPU, traffic, latency, errors, connections) for the resource.
  </Step>
  <Step title="Classify">
    Decides what kind of problem it is from the signals.
  </Step>
  <Step title="Read logs">
    Tails CloudWatch / Cloud Logging to find the actual errors.
  </Step>
  <Step title="Root cause + fix">
    Explains the cause and proposes a fix, applying it only after you confirm.
  </Step>
</Steps>

## Safety: confirm before changes

<Warning>
For anything **billable or destructive**, Nimbus shows a plan and an estimated cost and waits for your explicit approval before acting. Read-only questions (inventory, telemetry, logs, cost) run without confirmation.
</Warning>

## Watching it work

As Nimbus works, the chat streams its **internal reasoning** and each **tool call** (for example, running an AWS CLI command or reading logs) as a live timeline, followed by its written answer. Results also flow into the [Canvas](/docs/canvas), [Resources & Cost](/docs/resources-and-cost), and Overview, so the whole app reflects what just happened.

## Design vs Agent mode

The chat runs in **design** mode (Canvas-only, no real cloud) or **agent** mode (operates your real cloud). The Canvas view uses design mode for sketching; say "deploy" to switch to agent mode and provision for real. See [How Nimbus works](/docs/how-nimbus-works).
